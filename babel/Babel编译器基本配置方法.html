Babel编译器基本配置方法

1 概述

Babel是一个编译器，从大的层面来说，它有三个主要的步鄹解析、转义、代码生成。 这个过程和大多数编译器是一样的。 Babel所做的工作就是将我们所做的代码进行某些转译工作，然后输入我们想要的代码。它的最常用的功能是对新版JS语言(如ES6， ES7， ES8)及语言的新特性(如stage-x)转译成ES5语言，以适应低版本的浏览器。

Babel有常见的几种使用方式：

  1. 使用.babelrc文件
  2. 在package.json中配置
  3. 命令行

使用.babelrc文件的实例如下：

{
  "plugins": ["transform-react-jsx"],
  "ignore": [
    "foo.js",
    "bar/**/*.js"
  ]
}

使用package.json配置的方法如下：

{
  "name": "my-package",
  "version": "1.0.0",
  "babel": {
    // 在这里写配置
  }
}

Babel会在当前编译的文件所在的目录查找.babelrc文件，如果不存在，则会向上级目录中查找，直到找到一个.babelrc文件，或者是找到一个有着"babel": {}配置的package.json文件。

你可以在options中配置"babelrc": false来禁用查找行为，如果是在命令行下使用--no-babelrc。

2 .babelrc配置文件介绍

Babel的配置主要是通过项目下的.babelrc文件来完成的，它有两个非常重要的属性： presets和plugins。 如下面所示：

{
    "presets": [],
    "plugins": []
}

因为没有对它做任何的配置，所以对我们而言，它所做的工作就相当于const babel = code => code;, 也就是将代码原封不动地转译了一遍。

比如：

test.js

class Foo {
  method() {}
}

转译后的结果：

compiled.js
class Foo {
  method() {}
}

完全没变。

当然这并不是我们期望的，所以我们来看看presets和plugins的功能。

前面在讲解presets的时候已经对plugins有所涉及了，大体而言，plugins只是聚焦转译的某一方面，Babel官方列出了如下一些常用的plugins:

plugins的介绍到此结束，我们接下来看看下面两个问题：

  1. 同时指定了presets和plugins的时候它们调用的顺序是怎么样的呢？
  2. presets和plugins自身的调用顺序是怎么样的呢？
2.2.1 Plugin/Preset 顺序

对于前面两个问题，Babel有如下结论：

  • Plugins 运行在 Presets 之前.
  • Plugin 自身的顺序是从第一个到最后一个.
  • Preset 自身的顺序正好相反 (即，从最后一个到第一个).

比如，对于下面的例子：

{
  "plugins": [
    "transform-decorators-legacy",
    "transform-class-properties"
  ]
}

它将会先运行transform-decorators-legacy，再运行transform-class-properties。

而对于有如下配置的presets而言, 它按照stage-2, react, es2015顺序运行：

{
  "presets": [
    "es2015",
    "react",
    "stage-2"
  ]
}

这通常是为了更好的后向兼容而设计的，因为大多数用户会将es2015列在stage-0前面。

2.2.2 Plugin/Preset 的Options参数

大多数presets和plugins都会提供options参数用于更加具体的设置他们的工作方式。 要使用options参数，你需要采用嵌套数组的形式，每一个preset/plugin为一个单独的数组，数组的第一个索引存放preset/plugin名称，第二个索引位置存放options对象。 比如：

{
  "plugins": [
    ["transform-async-to-module-method", {
      "module": "bluebird",
      "method": "coroutine"
    }]
  ]
}

以及

{
  "presets": [
    ["es2015", {
      "loose": true,
      "modules": false
    }]
  ]
}

3.1 babel-preset-env

不带任何options的babel-preset-env的效果等同于babel-preset-latest(也等同于同时设置babel-preset-es2015, babel-preset-es2016, babel-preset-es2017)。 也就是说，下面的三种设置都是等效的：

{
  "presets": ["env"]
}


{
  "presets": ["babel-preset-latest"]
}


{
  "presets": ["babel-preset-es2015", "babel-preset-es2016", "babel-preset-es2017"]
}

当然，前提是你需要安装对应的包。

你也可以在babel-preset-env中指定只包含你想要支持的浏览器的polyfill和transforms。这样可以是你最终的代码体积减少一些。

下面这个例子包含的polyfill和transforms只会将源码转换成支持最近2个版本的浏览器，以及>=7的safari浏览器。 它使用的是browserslist 库来解析浏览器信息, 所以你可以使用任何browserslist支持的查询格式.

{
  "presets": [
    ["env", {
      "targets": {
        "browsers": ["last 2 versions", "safari >= 7"]
      }
    }]
  ]
}

类似的，如果你的目标是node而不是浏览器，你也可以像下面来配置：

{
  "presets": [
    ["env", {
      "targets": {
        "node": "6.10"
      }
    }]
  ]
}

你也可以像下面这样指定Babel将代码只编译成适应当前node版本的代码：

{
  "presets": [
    ["env", {
      "targets": {
        "node": "current"
      }
    }]
  ]
}

babel-preset-env的常用配置如下：

targets

值： { [string]: number | string }, 默认为 {}

用于设置环境和版本，我们推荐使用string，因为在某些有镜像版本的情况下适用，比如： `node: "6.10"`

一些可选的环境有chrome, opera, edge, firefox, safari, ie, ios, android, node, electron

targets.node

值： number | string | "current" | true
如果你想要编译成当前版本node所支持的语法，你可以指定为`"node": true` 或 `"node": "current"`, 这种设置和` "node": process.versions.node`效果一样.

targets.browsers

值： Array<string> | string

loose

值： boolean 默认false。
用于以一种松散模式转译

modules

"amd" | "umd" | "systemjs" | "commonjs" | false, 默认为 "commonjs"
将ES6模块语法转成指定的模块语法

useBuiltIns

值： boolean， 默认false
使用内置的babel-polyfill, 这个稍后再讲

3.4 babel-polyfill

babel-polyfill既不是preset也不是plugin，放在这里讲是因为，虽然Babel可以编译所有新的js语法，但是对于js API来说，并不是如此。 比如下面的例子：

test.js

function addAll() {
  return Array.from(arguments).reduce((a, b) => a + b);
}

.babelrc配置如下：

{
    "presets": [
        "es2015"
    ],
    "plugins": [
    ]
}

输出如下：

function addAll() {
  return Array.from(arguments).reduce(function (a, b) {
    return a + b;
  });
}

可以看到，它仅仅只是把箭头函数换成了普通函数，对于Array.from没有做任何的转换，但是Array.from在某些(大部分是IE)浏览器下是不支持的。所以我们使用babel-polyfill。

Babel 用了优秀的 core-js 用作 polyfill，并且还有定制化的 regenerator 来让 generators（生成器）和 async functions（异步函数）正常工作。使用babel-polyfill只需要在代码顶部引入即可：

import "babel-polyfill";

我们来看一下引入babel-polyfill后效果：

test.js

import "babel-polyfill";
function addAll() {
  return Array.from(arguments).reduce((a, b) => a + b);
}

.babelrc配置不变

输出如下：

function addAll() {
  return Array.from(arguments).reduce(function (a, b) {
    return a + b;
  });
}

可以看到，没有太大的差异，只是把ES6的模块换成了commonJS模块的用法。
