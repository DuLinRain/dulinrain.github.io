HTTP状态码中303与302、307有什么区别？

一、概述

我们在HTTP状态码中302和307有什么区别？中详细讨论了302和307的区别，实际上他们除了对Location指定资源的请求方式有区别外，没有其它的区别。今天呢我们再看看303的含义以及它和302/307的区别在哪里。

二、303

翻看HTTP协议（RFC2616），我们看到它对HTTP 303状态码描述如下：

这里面可以发现，大体上和302/307行为一样，都是在Location中指定资源地址，浏览器需要重新请求Location中指定资源的地址。但是归纳起来主要有3点需要注意：

  1. redirect的那个请求一定是GET，不管original请求类型是什么。
  2. 老的客户端可能不识别303，所以遇到这种情况可能会用302代替303。
  3. original请求一定不允许被缓存，但是redirect请求可以被缓存。

也就是说，最主要的区别体现在redirect请求类型以及缓存策略。

三、与302/307的区别

所以就第1点而言，302和303几乎是一样的，没什么区别，这也就是第2点可以用302代替303的原因。 也因此在这一点上，307和303/302的区别就体现出来了。

第二点涉及到缓存，我们之前讨论302和307区别的时候没有讨论这一点，这是因为他们2者在这一点上没有任何区别，我们把HTTP协议对他们的描述粘贴出来：

也就是说，它俩本身不可缓存，除非我们设置了Cache-Control或Expires头域让它缓存。

我们再看看303的缓存策略：

The 303 response MUST NOT be cached, but the response to the second (redirected) request might be cacheable.

也就是上面提到的第3点：original请求一定不允许被缓存，但是redirect请求可以被缓存。

我们下面不妨用几个例子来验证一下。

四、实例

4.1 302/307的缓存策略实例

我们先测试302/307在设置Cache-Control和不设置Cache-Control情况下的表现，server端代码如下：

var express = require('express');
var cors = require('cors')
var app = express();
app.use(cors())

app.all('/test', function (req, res) {
  console.log('original请求方式: ' + req.method)
  res.set({
    // 'Cache-Control': 'max-age=3600', //不缓存
    'Location': 'http://127.0.0.1:3000/test1'
  }).status(302).end('hehe1')
});
app.all('/test1', function (req, res) {
  console.log('redirect请求方式: ' + req.method)
  res.set({
    // 'Cache-Control': 'max-age=3600' //不缓存
  }).end('redirect请求')
});

var server = app.listen(3000, function () {
  var host = server.address().address;
  var port = server.address().port;

  console.log('Example app listening at http://%s:%s', host, port);
});

客户端代码如下：

我们不改变客户端代码，将服务端代码状态码改为303，

var express = require('express');
var cors = require('cors')
var app = express();
app.use(cors())

app.all('/test', function (req, res) {

五、总结

303、302、307的区别可总结如下：

  • 303和302在处理redirect请求类型时行为一致，即都会使用GET，307会使用与orginal相同的类型。
  • 302和307缓存策略一致，即不论是original还是redirect请求都可以被缓存，302 orginal请求不能缓存，redirect可以缓存。
