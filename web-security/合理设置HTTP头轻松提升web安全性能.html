合理设置HTTP头轻松提升web安全性能

本文将讲述用于提升web安全的一些最重要的HTTP头。我们将理解这里面每种HTTP是用来干嘛的以及当这些头被错误配置的时候会受到哪些攻击。

下面列出的是我们将要讲述的一系列的HTTP头，这其中又可大体上分为两大部分：

预防攻击的headers
  • HTTP Strict Transport Security
  • Content Security Policy
  • X-FrameOptions
  • X-XSS-Protection
  • X-Content-Type-Options
  • Access-Control-Allow-Origin
  • Set-Cookie
泄漏信息的headers
  • Server
  • X-Powered-By
  • X-AspNet-Version

一、 HTTP header的背景

HTTP头是客户端和服务端通过HTTP协议通信时传递的一些信息，当我们在浏览器输入一个地址或者点击一个链接的时候，web浏览器会发送包含HTTP客户端headers的HTTP请求，同样服务端的响应会包含服务端headers。

HTTP有数十个headers，对他们一个个的介绍并不是本文的目的。 本文只讨论对web安全有影响的HTTP Server headers。

二、 预防攻击的headers

2.1 HTTP Strict Transport Security (HSTS)

HTTP Strict Transport Security指示浏览器只能通过https去访问web服务器。 一旦在服务端进行了设置，HTTP响应头中将包含Strict-Transport-Security头字段。 当浏览器收到这个头的时候会将所有的请求通过https发送。 HSTS可设置为3个值：

  • Max-age： 该值定义了在多久的时间内，只能通过https访问，默认的取值是31536000秒，也就是1年(365天)，这也是该值的最大有效值。 服务端在每次响应的时候更新这个值这样该值就不会过期。
  • IncludeSubDomains： 表示将该设置同时应用于子域名。
  • Preload：Preload 列表是硬编码在Google Chrome浏览器里的，网站的拥有者可以将自己的URL提交给Chrome放进这个列表中。该列表由Google维护，但是也被其它浏览器使用，该列表可以在这里看到：https://chromium.googlesource.com/chromium/src/+/master/net/http/transport_security_state_static.json

攻击场景

如果未启用HSTS，攻击者可以执行中间人攻击并从用户的Web会话中窃取敏感信息。 想象一下，受害者连接到开放式Wi-Fi的情况实际上是在攻击者的控制之下。 通过HTTP访问网站将允许攻击者拦截请求并读取敏感信息。因为HTTP传输的都是明文。 如果该站点使用HTTPS，但用户使用HTTP访问它，会将其重定向到HTTPS。 如果同一用户之前访问过该网站，则浏览器中记录的HSTS详细信息会导致通过HTTPS自动建立连接。

2.2 Content Security Policy（CSP策略）

Content Security Policy指示浏览器只加载被策略定义所允许的内容，该策略使用的是白名单的方式来告诉浏览器只能从哪里加载图片、js脚本、CSS等资源，如果配置得当，该策略可以阻止网站遭受跨站脚本攻击(XSS)、点击劫持（ClickJacking）以及HTML注入( HTML injection )攻击。

该HTTP头就叫做Content-Security-Policy，可以设置的指令包括default-src, script-src, media-src, img-src。 这些指令指示浏览器从哪里加载这些类型的资源。

下面是Content-Security-Policy设置的一个例子：

Content-Security-Policy: default-src 'self'; media-src media123.com media321.com; script-src script.com; img-src *;

浏览器会做如下解释：

  • default-src 'self': 只能从当前域名下加载所有资源。
  • media-src media123.com media321.com: 只能从media1.com 和 media2.com加载媒体资源。
  • script-src script.com: 只能从 script.com 加载脚本资源
  • img-src *: 可以从任意地方加载图片。

2.3 X-FrameOptions

这个头也是用来阻止用户遭受点击劫持攻击，它是通过禁止浏览器通过iframe加载该页面实现的。 X-FrameOptions可以设置3个值：

  • X-Frame-Options: DENY 这会阻止该页面被任何网站的iframe加载，包括他自己所属的网站。
  • X-Frame-Options: sameorigin 这只允许和该页面同域的网站通过iframe加载该页面。 也就是说www.site.com下的页面只允许在www.site.com域名下被iframe加载。
  • X-Frame-Options: allow-from uri 只允许在指定的域名/uri下通过iframe加载该页面。

攻击场景

攻击者可以欺骗用户访问恶意网站，恶意网站会将目标应用程序加载到不可见的iframe中。 当用户点击恶意应用程序（例如基于网络的游戏）时，点击将被“窃取”并发送到目标应用程序（Clickjacking）。 因此，用户将在未经他同意的情况下点击应用程序，这可能导致执行一些未预期的操作（例如删除帐户等）。

2.4 X-XSS-Protection

这个头是用来阻止跨站脚本攻击（XSS）的，它在现代浏览器中与XSS Filters一起工作，并且可以设置三种模式：

-X-XSS-Protection: 0; - 表示禁用 XSS filter

  • X-XSS-Protection: 1; -启用 XSS filter, 如果检测到XSS攻击，浏览器将清理页面内容以阻止脚本执行。
  • X-XSS-Protection: 1; mode=block - 如果检测到XSS攻击将阻止页面渲染。

2.5 X-Content-Type-Options

这个响应头是为了阻止MIME嗅探（ MIME sniffing）攻击。什么是MIME嗅探？ MIME嗅探是Web浏览器的一项功能，用于检查所服务文件的内容。 它的工作原理如下：

  1. Web浏览器请求文件。 服务器发送带有HTTP标头Content-Type集的文件。
  2. Web浏览器“嗅探”此文件的内容以确定文件格式。
  3. 完成分析后，浏览器会将其结果与服务器发送的结果进行比较。 如果不匹配，浏览器将使用标识的格式。

这会导致一种安全攻击，如何做到？

攻击场景
  1. 应用程序允许用户上传图像文件并验证其扩展名
  2. 用户上传带有jpg或png扩展名的图像文件，但此文件也包含恶意HTML代码
  3. 浏览器使用包含代码的HTML呈现文件并在浏览器中执行

  • Content-Disposition: 在下载pentest.html文件的时候会强制浏览器出弹窗。
Content-Disposition: attachment; filename=pentest.html
  • X-Download-Options: 当被设置为 noopen 时, 会强制用户在打开文件前先将其保存成本地文件, 而不是在浏览器中直接打开。

2.6 Access-Control-Allow-Origin

Access-Control-Allow-Origin其实是CORS头，通常用于解决跨域问题。 这个头允许指定的非同域站点去访问给定的资源。 具体的使用方法很简单，这里就不再赘述，但是需要注意的是区分简单/非简单CORS请求的不同，也就是要不要对OPTIONS响应做设置。

攻击场景

如果Access-Control-Allow-Origin配置不当，攻击者可以使用其他第三方网站从目标网站读取数据。 许多开发人员使用通配符访问Access-Control-Allow-Origin头，这样设置会导致允许任何网站从他们的网站读取数据。

2.7 Set-Cookie

应用程序设置的cookie值由服务器在Set-Cookie头中发送。 收到此头后，浏览器将在Cookie头中发送每个HTTP请求的cookie。

HTTP cookie通常包含敏感信息（尤其是session cookie），需要保护它们以防止未经授权的访问。

可以通过设置以下属性来保护cookie：

  • Secure：设置该属性后， cookie将仅通过HTTPS发送，而不是通过明文HTTP协议发送
  • HTTPOnly：设置该属性后，浏览器不允许通过JavaScript代码访问使用此属性设置的cookie的内容。 这有助于减少通过XSS攻击实现的会话劫持。

结论

可以在服务器上配置HTTP标头，以增强Web应用程序的整体安全性。 这些头不会使应用程序本身更安全，但它们会阻止利用应用程序的潜在漏洞。
