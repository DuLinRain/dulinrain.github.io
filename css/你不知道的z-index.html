你所不知道的z-index

一、概述

z-index是CSS里面经常会用到的一个属性，看到这个标题时，可能你会觉得z-index那么简单有什么好说的，但可能你过于乐观了，z-index其实相当复杂，你所了解的也许只是冰山一角而已。

举个例子，对于下面的HTML和CSS，你觉得红、绿、蓝三个颜色块的堆叠效果如何呢？

<div>
  <span class="red">Red</span>
</div>
<div>
  <span class="green">Green</span>
</div>
<div>
  <span class="blue">Blue</span>
</div>

.red, .green, .blue {
  position: absolute;
  width: 100px;
  color: white;
  line-height: 100px;
  text-align: center;
}

.red {
  z-index: 1;
  top: 20px;
  left: 20px;
  background: red;
}

.green {
  top: 60px;
  left: 60px;
  background: green;
}

.blue {
  top: 100px;
  left: 100px;
  background: blue;
}

想必大家都知道，它的结果会如下，因为都是绝对定位，然后只有red设置了正的z-index，而green和blue的显示则与他们在文档中的顺序有关：

那么，如果现在让你想办法将red方块放到最下面，并且保证：

  1. 不要动HTML结构及位置
  2. 不要改变任何元素的z-index
  3. 不要改变任何元素的position属性

元素的其它css属性可以任意改动，那么你可以实现吗？

想必大家此时便会犯难了。其实解决方法很简单，就是给第一个div元素加一个非1的opacity属性：

div:first-child {
  opacity: .99;
}

那么还有没有其它的方法呢？有，比如下面2种：

div:first-child {
    transform: translateZ(0)
}

div:first-child {
    filter:blur(0)
}

那么这样做为什么可行呢？这只是一个csshack还是有原理的呢？实际上它与我们接下来要讲的 “stacking order” 有关。

二、Stacking Order 栈顺序

所谓Stacking Order，就是指HTML文档中的元素，要么出现在别的元素的前面，要么出现在别的元素的后面。 这个在没有z-index和position属性的时候是相当简单的。因为此时，元素的Stacking Order就是他们在文档中出现的顺序来。

但是，当你使用了position属性后，任何定位元素以及它的子元素都会出现在非定位元素的前面。

注意，这里的非定位元素是指没有position属性或position属性为static的元素，而定位元素是指position属性为非static的元素。

当使用了z-index属性之后，情况变得又复杂了一点。 我们很自然的会认为，具有较大z-index值的元素位于具有较小z-index值的元素前面，并且具有z-index的任何元素都位于没有z-index的任何元素之前，但现实情况并不是这么简单。 因为有以下2点需要关注：

  1. z-index只有用在定位元素(position属性非static)上才有效。
  2. z-index可以创造stack context。

现在情况变得更复杂了，而且还多了另一个概念Stack Context。 下面我们先来看看Stack Context。

三、Stack Context 栈上下文

具有共同父级的元素组在堆叠顺序中一起向前或向后移动构成了所谓的堆叠上下文。 充分理解堆栈上下文是真正掌握z-index和堆栈顺序如何工作的关键。

每一个Stack Context都有一个单一的HTML元素作为这个Stack Context的根元素。 当一个新的Stack Context被创造时，那么该Stack Context根元素的所有孩子元素都将处于该Stack Context中，这意味着，如果有A、B 两个Stack Context，A在B前面，那么不管你如何操作A的孩子元素的Stacking Order，这个孩子元素都不可能跑到B后面去。 因为无论你如何操作A的孩子的Stacking Order，他们都在A的Stack Context中移动。

那么新的Stack Context是如何产生的呢？它们有如下3种方式：

  1. 该元素是整个文档的根元素。也就是说该元素是html元素。
  2. 该元素有一个非static的position属性以及一个z-index不为auto的属性。
  3. 该元素有一个小于1 的opacity属性。

想必前2条大家可能或多或少都知道，但是对于最后一条的存在可能根本不知道。这也就是我们在开篇使用的技巧。

注意：除了opacity属性，一些较新的css属性，如 transforms, filters, css-regions, paged media等属性也会产生Stack Context。 我们比较常见和常用的主要是opacity, transforms, filters这几个。

现在了解了Stack Context和Stacking Order，那么我们如何决定一个元素在堆栈中的位置呢？我们继续分析。

四、 如何决定元素在Stacking Order的位置？

要确定一个元素在整个文档堆栈中的位置稍显复杂，我们先看看在相同Stack Context中元素的Stacking Order。

4.1、 相同Stack Context中元素的Stacking Order

假设有很多个元素都在同一个Stack Context中，那么他们的Stacking Order是如何确定的呢？下面是基本的规则（从后往前）：

  1. Stack Context的根元素。
  2. 具有负z-index值的定位元素（及其子元素）（较大的z-index值堆叠在较小值之前;具有相同值的元素根据HTML中的顺序堆叠）
  3. 非定位元素（按HTML中的呈现顺序）
  4. 定位元素（及其子元素）的z-index值为auto（通过HTML中的呈现顺序）
  5. 具有正z-index值的定位元素（及其子元素）（较大的z-index值堆叠在较小值的前面;具有相同值的元素根据HTML中的呈现顺序堆叠）

注意：具有负z-index值的定位元素首先在堆叠上下文中排序，这意味着它们出现在所有其他元素的后面。 因此，一个元素可能出现在它自己的父亲后面，这通常是不可能的。 这仅在元素的父元素位于相同的堆叠上下文中且不是该堆叠上下文的根元素时才有效。

4.2、全局Stacking Order

在了解了新的Stack Context如何/何时产生以及在Stack Context中掌握Stacking Order的情况下，确定特定元素在全局Stacking Order中出现的位置并不是那么难了。

避免判断错误的关键是能够发现何时形成新的Stack Context。 如果你为一个元素设置了一个很大的z-index，然而它没有按照Stacking Order向前移动，那么查看它的祖先树并查看它的父节点是否有任何形式的堆栈上下文。 如果他们产生了Stack Context，那么你很大的z-index的值对你没有任何帮助。

现在，回过头来看看开篇的例子，显然那并不是css hack，而是确实有依据的。

当我们没有给第一个div设置opacity的时候，它的Stacking Order如下：

<div><!-- 1 -->
  <span class="red"><!-- 6 --></span>
</div>
<div><!-- 2 -->
  <span class="green"><!-- 4 --><span>
</div>
<div><!-- 3 -->
  <span class="blue"><!-- 5 --></span>
</div>

此时此刻，文档中有2个Stack Context，根元素生成的和.red元素生成的。

对于非定位元素，按他们在文档中出现的顺序（规则3），对于定位元素但z-index=auto的元素，也是按在文档中的顺序（规则4），对于定位元素且z-index为正，按z-index值得大小（规则5）。

但是当，我们给第一个div设置了小于1的opacity，那么这个div会形成一个Stack Context。它的子元素都会在该Stack Context中排序。这时的情况如下：

<div><!-- 1 -->
  <span class="red"><!-- 1.1 --></span>
</div>
<div><!-- 2 -->
  <span class="green"><!-- 4 --><span>
</div>
<div><!-- 3 -->
  <span class="blue"><!-- 5 --></span>
</div>

这下就很容易理解了，对于非定位元素（三个div），按他们在文档中出现的顺序（规则3），对于定位元素但z-index=auto的元素，也是按在文档中的顺序（规则4），对于定位元素且z-index为正，按z-index值得大小（规则5），但是由于它的父亲生成了一个新的Stack Context，所以它的z-index只是相对于这个新Stack Context，所以，不管z-index设置多大，red块都会出现在其它两个块的下面。

五、结论

z-index看似简单，其实结合position和opacity等属性一起就很复杂了，我们在使用的时候一定要把各种情况考虑周全。
